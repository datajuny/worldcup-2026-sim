# Dynamic Workflows 월드컵 예측 사이트 생성 프롬프트

아래 내용을 Claude Code에 그대로 넣는다. 빈 폴더에서 시작하는 것을 전제로 한다.

## 실행 전 화면 체크

- `claude --version`으로 Claude Code 버전을 확인한다.
- 필요하면 `claude update`로 업데이트한다.
- Claude Code v2.1.154 이상인지 확인한다.
- Claude Code에서 `/config`를 열고 `Dynamic workflows` 항목이 `ON`인지 확인한다.
- 실제 작업 입력은 `/effort ultracode`로 시작한다.

```text
/effort ultracode

한국의 2026 월드컵 성적과 우승팀을 데이터 기반으로 예측하는 정적 웹사이트를 처음부터 만들어줘.

목표는 그럴듯한 스포츠 예측 글을 쓰는 게 아니야.
공개 데이터를 다각도로 조사하고, 그걸로 통계 모델을 세우고, 1만 번 시뮬레이션하고, 결과를 검증하고, GitHub Pages에 배포 가능한 시각화까지 한 번에 끝내는 거야.

이건 여러 단계가 얽힌 복잡한 작업이야.
팀을 내가 짜주지 않을 테니, 아래 작업들이 다 필요하다는 걸 보고 네가 알아서 sub-agent로 나눠서 병렬로 처리해줘.

최종 산출물은 GitHub Pages에 바로 올릴 수 있어야 해.
루트에 index.html을 만들고, 서버가 필요한 기능은 넣지 마.
CSS와 JavaScript는 index.html에 포함하거나 상대 경로로 연결하고, 로컬 절대 경로는 쓰지 마.

────────────────────────────
[작업 1] 자료조사 — ★ 랭킹 하나로 끝내지 마

기본 정보.
- 공식 대회 포맷, 12개 조 편성, 개최국(미국·캐나다·멕시코)을 공식 출처에서 조사해.
- 2026 월드컵은 12개 조 4팀 체제야. 각 조 1·2위 24팀에 조 3위 성적 상위 8팀이 더해져 32강에 진출해.
- 조 3위 상위 8팀은 FIFA 랭킹만으로 고르지 마. 공식 규정대로 승점 → 골득실 → 득점 → 팀 conduct score → FIFA 랭킹 순으로 비교해.
- 32강부터는 다시 조별리그를 하지 않고 단판 토너먼트로 진행해.

전력 데이터 (★ 여기가 핵심).
FIFA 랭킹 포인트 하나만 쓰면 진짜 시뮬레이션이 아니야. 랭킹이 비슷한 팀들이 구별이 안 돼서 결과가 조 편성 운으로만 갈려.
그러니 우승 경쟁권 주요 팀만 아래를 추가로 조사해. ★ 토큰을 아끼려고 범위를 좁힌다 — 상위 8개국만, 항목도 두 개만.
- 최근 10경기 결과 (승/무/패) → 현재 폼 지표
- 스쿼드 시장가치 (Transfermarkt 등 공개 추정치) → 선수단 전력
- (토큰 여유가 있을 때만) 핵심 선수 부상·결장 여부

수집 규칙.
- ★ 팀당 WebSearch/WebFetch는 1~2회로 제한해. 한 번 검색으로 폼과 시장가치를 같이 찾아.
- 모든 데이터는 출처 URL과 수집 날짜를 sources.md에 남겨.
- 공식·신뢰도 높은 출처를 우선하고, 약한 출처는 신뢰도를 함께 표시해.
- 못 구한 데이터는 추정하지 마. "미확보"로 표시하고, 그 팀은 랭킹 기반으로만 처리해.
- 상위 8개국을 뺀 나머지 40개국은 FIFA 랭킹 기반으로만 처리해. 폼·시장가치 조사하지 마.

────────────────────────────
[작업 2] 데이터 과학 모델링 — ★ 랭킹 비슷한 팀을 변별해

- 팀 전력 점수를 FIFA 랭킹 하나로 만들지 마.
- FIFA 랭킹을 베이스로 하되, 최근 폼·부상·스쿼드 시장가치를 가중치로 반영한 합성 전력 점수를 설계해.
- ★ 랭킹 포인트가 거의 같은 상위권 팀들(예: 1~3위가 몇 점 차)은, 폼·부상·전력으로 반드시 구별되게 해. 랭킹 동률이 우승 확률 동률로 이어지면 안 돼.
- 각 보정 항목의 가중치와 그 근거를 모델 문서에 명시해. 설명 가능한 모델이어야 해.
- 경기별 득점은 이 합성 전력 점수 차이를 바탕으로 Poisson 분포 또는 그에 준하는 확률 모델로 계산해.
- 모델 선택 이유, 가중치, 반영하지 못한 변수를 명시해.

────────────────────────────
[작업 3] 시뮬레이션 구현

- Python으로 scripts/simulate.py를 작성해.
- 합성 전력 점수를 입력으로 월드컵 전체를 최소 10,000회 Monte Carlo 시뮬레이션해.
- 랜덤 시드를 고정해서 결과가 재현되게 해.
- 결과는 data/worldcup-2026.json으로 저장해.
- JSON에는 우승 확률, 라운드별 진출 확률, 한국 결과 분포, 한국 조별 경기별 승무패 확률, 조별리그 결과, 토너먼트 결과, 사용한 전력 점수와 그 구성, 모델 한계를 포함해.

────────────────────────────
[작업 4] 검증 — 다른 작업 결과를 의심하고 따져

최소한 다음을 확인해.
- 대회 포맷과 진출 규칙이 공식 규칙과 맞는지.
- 사용한 데이터 출처가 sources.md에 전부 남아 있는지.
- 랜덤 시드가 고정되어 재현 가능한지.
- 경기당 평균 골 수가 실제 월드컵 평균(약 2.5~2.8골)과 크게 어긋나지 않는지.
- ★ 전력 점수가 FIFA 랭킹만이 아니라, 최근 폼·부상·시장가치를 실제로 반영했는지.
- ★ 랭킹이 비슷한 상위권 팀들이 전력 점수에서 변별되는지. (안 그러면 자료조사·모델링으로 되돌려 보내)
- 한국 결과 분포와 JSON 수치가 화면에 표시되는 수치와 일치하는지.
- 결과를 단정적 예측이 아니라 확률 분포로 설명하는지.

검증을 통과하지 못한 내용은 사이트에 넣지 말고, 수정하거나 한계로 명시해.

────────────────────────────
[작업 5] 시각화

- data/worldcup-2026.json을 바탕으로 index.html을 만들어. 외부 서버 없이 브라우저에서 바로 열리는 정적 HTML이어야 해.
- 모바일에서도 읽기 좋아야 하고, 첫 화면에서 다음이 바로 보여야 해.
  - 우승 확률 TOP 10.
  - 한국의 성적 분포.
  - 한국 조별 경기별 승무패 확률.
  - ★ 각 팀 전력 점수가 어떤 데이터(랭킹·폼·부상·시장가치)로 구성됐는지 한눈에 보이는 패널.
  - 다크호스 또는 의외의 결과.
  - 이 모델을 믿으면 안 되는 이유와 한계.
- 디자인은 미드나잇 로그 톤으로. 어두운 배경, 선명한 숫자, 큰 그래프, 모바일 가독성 우선. 화려한 장식보다 데이터가 먼저 보이게.

────────────────────────────
[작업 6] 배포 문서화

README.md와 sources.md를 작성해.
README.md에는 다음을 넣어.
- 로컬에서 실행하는 방법.
- GitHub Pages에 배포하는 방법.
- 어떤 데이터를 썼는지 (랭킹·폼·부상·시장가치 각각).
- 모델이 어떤 가정과 가중치를 썼는지.
- 이 결과를 실제 예측으로 믿으면 안 되는 이유.
- 다른 주제에 이 워크플로우를 재사용하는 방법.

────────────────────────────
작업 규칙.
- 각 작업은 독립적으로 판단해. 검증은 반드시 다른 결과를 의심하고 확인해.
- 모르는 수치는 만들지 마. 스포츠 칼럼식 표현보다 데이터·가정·한계·확률 분포를 우선해.

최종 파일 구조.
- index.html
- data/worldcup-2026.json
- scripts/simulate.py
- README.md
- sources.md
- .nojekyll

처음에 10줄 이내로 전체 실행 계획을 보여준 뒤, 바로 구현까지 진행해.

★ 토큰 절약 (중요 토큰 여유가 많지 않아).
- sub-agent는 꼭 필요한 만큼만 띄워. 같은 일을 여러 agent가 중복해서 하지 않게 해.
- 자료조사는 상위 8개국, 팀당 검색 1~2회로 제한해. 나머지 40개국은 FIFA 랭킹 기반으로만.
- 검증은 한 번만. 통과 못 한 항목만 고치고, 무한 재검토 루프는 돌지 마.
- 중간 설명이나 요약을 길게 쓰지 말고, 결과물 위주로 진행해.
- 시뮬레이션 1만 회 자체는 로컬 Python 실행이라 토큰을 거의 안 쓰니, 횟수는 줄이지 않아도 돼.
- 전체 작업을 최대한 적은 토큰으로 끝내는 걸 우선순위로 삼아.
```
